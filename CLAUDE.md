## 2. Стек проекта

### Текущий стек (уже работает)

- **Backend:** Python 3.11+ / Django 5.x
- **БД:** PostgreSQL 15+
- **WebSocket:** Django Channels 4 + Daphne (сейчас `InMemoryChannelLayer`, заменим на Redis)
- **Шаблоны:** Django Template Language + Bootstrap 5 + django-crispy-forms
- **Документы:** python-docx (договоры), openpyxl + reportlab (экспорт XLSX/PDF)
- **Авторизация:** кастомная модель `User` на UUID, 4 роли (client / owner / manager / administrator)
- **Конфиги:** python-decouple через `.env`

### Целевой стек (что добавляем под диплом)

- **REST API:** Django REST Framework + SimpleJWT (httpOnly cookie)
- **Frontend:** React 18 + Vite + TypeScript + TanStack Query + Tailwind CSS
- **Контейнеризация:** Docker + docker-compose (web / db / redis / mongo / nginx)
- **Redis:** channel layer для Channels вместо InMemory
- **MongoDB:** хранение чат-сообщений и логов (закрывает NoSQL-требование курсача)
- **pgvector:** расширение Postgres для embedding-поиска по описаниям инвентаря
- **AI:** OpenAI API через `langchain` для парсинга естественно-языковых запросов и эмбеддингов
- **Геолокация:** Yandex Maps JS API 2.1 + Yandex Geocoder (РФ-фокус)

## 4. Цели диплома

### Главное (must-have, делаем точно)

1. **AI-геолокационный поиск** — пользователь пишет «недорогие беговые лыжи в Казани на завтра», LLM парсит запрос, Yandex Geocoder даёт координаты, pgvector делает семантический поиск, фронт показывает результат на карте Yandex Maps. **Это центральная фишка диплома.**

2. **Расширение на всю РФ** — сейчас образно «один магазин». Делаем:
   - Модель `City` (справочник городов РФ)
   - Модель `PickupPoint` (точка выдачи: адрес, координаты, привязка к городу/менеджеру/владельцу)
   - У `Inventory` появляется `pickup_point`
   - Фильтр инвентаря по городу и радиусу от точки

3. **NDA-соглашение при регистрации** — отдельный чекбокс «согласен на обработку паспортных данных» (152-ФЗ), отдельная модель `PassportNDA` с датой принятия и версией документа.

4. **Docker для всех сервисов** — `docker-compose.yml` поднимает: Django+Daphne, PostgreSQL+pgvector, Redis (channel layer), MongoDB (логи и чаты). На защите запускается одной командой `docker-compose up`.

5. **REST API на DRF** — сериализаторы для всех ключевых моделей, `ViewSet`'ы, JWT в httpOnly cookie. Старые Django-views остаются для админ-панели.

6. **React-фронт** — Vite + TypeScript. Минимум страниц: каталог с фильтрами и картой, детали инвентаря, форма аренды, профиль, чат, регистрация/логин. Авторизация через JWT в httpOnly cookie (закрывает «никаких localStorage для уязвимых данных»).

7. **NoSQL хранилище** — MongoDB для чат-сообщений и application-логов (закрывает требование NoSQL из курсача и хорошо смотрится для диплома).

### Опциональное (если будет время)

- AI-помощник для описания инвентаря (владелец пишет «горный велосипед» → LLM генерирует описание)
- AI-модерация отзывов (фильтр спама/мата перед публикацией)
- Recommendation system (на основе истории аренд клиента)
- Светлая цветовая схема (Tailwind + custom palette)

### НЕ делаем (явно отложено пользователем)

- ❌ Микросервисная архитектура с брокером сообщений
- ❌ HTTPS-сертификаты (для production)
- ❌ Балансировка нагрузки
- ❌ SMTP-уведомления / Telegram-бот
- ❌ Восстановление пароля через email/SMS/QR
- ❌ Сжатие файлов (gzip в nginx)
- ❌ Production-деплой куда-либо

Если в задаче появляется что-то из этого списка — **спроси у пользователя, точно ли надо**, прежде чем делать.

## 5. Архитектурные принципы

### DRY, SOLID — соблюдай явно

- Дублирующиеся проверки (`if request.user.role != 'owner'`) выноси в декораторы (`@role_required('owner')`)
- Дублирующаяся валидация (например, `clean_account_number` в двух формах) — в отдельный validator
- Бизнес-логика — в `services/` каждого приложения, не в views и не в models

### Паттерны проектирования (нужен 1+ для защиты)

Для диплома достаточно реализовать **один из них явно** и уметь про него рассказать:

- **Стратегия** для расчёта рейтинга: `InventoryRatingStrategy`, `UserRatingStrategy` в `reviews/services/rating.py`
- **Наблюдатель** через Django signals: `post_save` на `Review` → пересчёт рейтингов (сейчас делается вручную в views)
- **Цепочка обязанностей** для пайплайна модерации инвентаря: `pending → moderation → contract_signed → published`

Рекомендую **Наблюдателя** — он наиболее естественно ложится на проект и реально упрощает код.

### Защита от уязвимостей

- ORM-параметризация (никаких `.raw()` SQL с интерполяцией строк) → SQL-инъекции
- `{% csrf_token %}` в формах (Django делает по умолчанию) → CSRF
- Экранирование в шаблонах (по умолчанию у Django, не использовать `|safe` без причины) → XSS
- JWT в httpOnly + Secure + SameSite cookie (НЕ в localStorage)
- CORS-headers разрешает только домен фронта

### 3 нормальная форма

В моделях в основном соблюдается, но **есть дубль** `User.loyalty_points` и `Client.loyalty_points`. **Не плодить такие дубли** в новых моделях.

### Запросы к БД

- `select_related()` для FK и OneToOne — обязательно когда обращаешься к связанным
- `prefetch_related()` для обратных связей и M2M
- `.only('field1', 'field2')` для горячих запросов где не нужны все поля (требование курсача)
- `.values()` или `.values_list()` для агрегированных данных

## 6. Правила работы (важно)

### Перед большим изменением — покажи план

Если задача затрагивает 3+ файлов или меняет архитектуру:
1. Сначала покажи **список файлов**, которые ты создашь или изменишь, **без кода**.
2. Кратко опиши, **что в каждом будет**.
3. Дождись `OK` от пользователя.
4. **Только потом** пиши код.

### Маленькие задачи — делай сразу

Правка одной функции, исправление бага в одном файле, добавление поля в модель — без плана, сразу к делу.

### Миграции

- Каждое изменение моделей = новая миграция через `python manage.py makemigrations`
- Имя миграции должно описывать суть: `0007_add_city_and_pickup_point.py`
- Не объединяй миграции вручную без необходимости
- Если миграция сложная (data migration) — пиши с RunPython

### Тесты

- Для новой бизнес-логики — пиши тест в `tests.py` соответствующего приложения
- Тесты Django: `python manage.py test`
- Для AI-функций — мокать LLM-вызовы (не дёргать реальный API в тестах)

### Коммиты

- Conventional Commits: `feat:`, `fix:`, `refactor:`, `docs:`, `test:`, `chore:`
- На английском, кратко
- После каждой логически завершённой задачи — коммит
- Push в ветку `master`

### Комментарии в коде

- На русском, как в существующем коде
- Не повторять что код делает (`# Сохраняем пользователя` над `user.save()` — мусор)
- Объяснять **почему** так, а не **что** делается
- Docstring для всех публичных функций и классов

### Когда не уверен — спроси

Не угадывай требования. Если в задаче неоднозначность — задай вопрос пользователю **один**, конкретный, с вариантами ответа.

## 7. Команды (как запускать)

### Локально (без Docker)

```bash
cd sportrent/
python -m venv venv
venv\Scripts\activate          # Windows
pip install -r ../requirements.txt
python manage.py migrate
python manage.py populate_db   # тестовые данные
python manage.py runserver
```

### Через Docker (после Недели 1)

```bash
docker-compose up -d           # поднять все сервисы
docker-compose exec web python manage.py migrate
docker-compose exec web python manage.py populate_db
docker-compose logs -f web     # смотреть логи
docker-compose down            # остановить
```

### Тесты

```bash
python manage.py test                              # все
python manage.py test users                        # конкретное приложение
python manage.py test users.tests.AuthServiceTest  # конкретный класс
```

### Заполнение демо-данных

```bash
python manage.py populate_db   # уже существует, не переписывать
```

## 8. Стиль кода

- **Python:** PEP 8, длина строки 100 символов (не 79)
- **Imports:** стандартные → сторонние → локальные, через пустую строку
- **Naming:** snake_case для функций/переменных, PascalCase для классов
- **Строки:** одинарные кавычки `'...'`, двойные `"..."` только для строк с апострофом
- **Django:** строго следуй конвенциям Django (app structure, model naming, URL naming)
- **React** (когда дойдём): функциональные компоненты, hooks, TypeScript-типы для props
- **Никаких** `print()` для отладки в коммитах — используй `logger.debug()`
